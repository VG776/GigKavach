## The 5-Step Onboarding Flow

```
Entry Point: Worker sends "JOIN"
│
├─ Step 0: Language Selection
│  Message: "👋 Welcome! Choose language (1-5)"
│  Response: Language code stored in Redis state
│  Next: ask_platform
│
├─ Step 1: Platform Selection  
│  Message: "1️⃣ Zomato, 2️⃣ Swiggy"
│  Response: Platform stored (zomato|swiggy)
│  Next: ask_shift
│
├─ Step 2: Shift Selection
│  Message: "1️⃣ Morning (6-2), 2️⃣ Day (9-9), ..."
│  Response: Shift stored (morning|day|night|flexible)
│  Next: ask_upi [SKIPS verification (step 3) in sandbox]
│
├─ Step 4: UPI Collection
│  Message: "Enter your UPI ID (e.g. ravi@upi)"
│  Validation: Must contain @ symbol
│  Response: UPI stored
│  Next: ask_pincode
│
├─ Step 5: Pin Codes Collection
│  Message: "Enter 1-5 pin codes (comma-separated)"
│  Validation: Each must be 6 digits
│  Response: Pins array stored
│  Next: ask_plan
│
└─ Step 6: Plan Selection [COMPLETION]
   Message: "1️⃣ Basic (₹69), 2️⃣ Plus (₹89), 3️⃣ Pro (₹99)"
   Response: Plan stored
   Action: CREATE worker record in Supabase
   Action: CREATE policy (coverage starts tomorrow)
   Action: SEND onboarding_complete message
   Action: CLEAR onboarding state from Redis
   Result: Worker account active, ready for disruptions
```

## Handler Implementation Reference

### 1. `handle_join(phone, message)` → str
**Purpose**: Start onboarding for new worker
**Flow**: 
- Check if worker already registered (Supabase query)
- If yes → return "already_onboarded" message
- If no → initialize Redis state with step=0, language="en", send "welcome" message
**Returns**: Language selection prompt

**Example**:
```
User: JOIN
Bot: 👋 Welcome to GigKavach! Reply with your language:
     1️⃣ English
     2️⃣ ಕನ್ನಡ
     ...
```

### 2. `route_message(phone, body)` → str
**Purpose**: Main dispatcher - routes incoming message to correct handler based on keyword OR onboarding step
**Logic**:
1. Parse keyword from message (first word, uppercase)
2. If keyword in [JOIN, STATUS, RENEW, SHIFT, LANG, HELP, APPEAL] → call corresponding handler
3. Otherwise, check if worker in onboarding state (Redis):
   - If state exists with step=0,1,2,4,5,6 → call step-specific handler
   - Otherwise → return HELP (default)
**Returns**: Response text for Twilio TwiML

### 3. `handle_status(phone, message)` → str
**Purpose**: Show worker's current coverage and zone disruption level
**Data Retrieved**:
- Worker record from Supabase (phone, language, pin_codes, plan)
- Active policy from Supabase (plan type, status)
- DCI score (mocked as 45 in sandbox)

**Example**:
```
User: STATUS
Bot: 📊 Zone 560001 Status:
     DCI Score: 45 (low)
     Coverage: Shield Plus ✅
     Shift: Morning
     Type HELP for all commands.
```

### 4. `handle_renew(phone, message)` → str
**Purpose**: Extend coverage for next week
**Actions**:
- Get worker from Supabase
- Create new policy record (week_start = tomorrow)
- Set coverage_pct and weekly_premium based on plan
- Send confirmation message in worker's language

**Example**:
```
User: RENEW
Bot: ✅ Renewed for next week! Your plus plan is active from 2024-12-18.
```

### 5. `handle_shift_update(phone, message)` → str
**Purpose**: Update worker's shift hours
**Format**: `SHIFT <1-4>` where 1=morning, 2=day, 3=night, 4=flexible
**Actions**:
- Parse choice from message
- Update workers.shift in Supabase
- Send confirmation

**Example**:
```
User: SHIFT 2
Bot: ✅ Shift updated to Day!
```

### 6. `handle_language_change(phone, message)` → str
**Purpose**: Switch worker's language preference
**Format**: `LANG <1-5>` where 1=en, 2=kn, 3=hi, 4=ta, 5=te
**Actions**:
- Parse language choice
- Update workers.language in Supabase
- Send confirmation in new language

**Example**:
```
User: LANG 2
Bot: ✅ ಭಾಷೆ ಕನ್ನಡಕ್ಕೆ ಬದಲಾಗಿದೆ!
```

### 7. `handle_help(phone, message)` → str
**Purpose**: Show all available commands
**Returns**: Multilingual command menu
**Message Key**: `MESSAGES["help"][lang]`

**Example (English)**:
```
GigKavach Commands:
• STATUS — Zone DCI & coverage
• RENEW — Renew for next week
• SHIFT — Update working hours
• LANG — Change language
• APPEAL — Contest a decision
• HELP — Show this menu
```

### 8. `handle_appeal(phone, message)` → str
**Purpose**: Open appeal/dispute window for fraud flags or payout issues
**Actions**:
- Get worker from Supabase
- Create appeal record with:
  - reason = message body
  - status = "open"
  - expires_at = now + 48 hours
- Send confirmation message

**Example**:
```
User: APPEAL I believe I was incorrectly flagged
Bot: ✅ Appeal submitted. We'll review within 48 hours and get back to you.
```

## State Management

### Redis Keys (Onboarding Only)
Format: `onboarding:{phone}`

**Example state after Step 2**:
```json
{
  "phone": "+919876543210",
  "step": 2,
  "started_at": "2024-12-17T10:30:45.123456",
  "language": "en",
  "platform": "zomato"
}
```

**TTL**: 3600 seconds (1 hour) - worker must complete or restart next day

### Supabase Persistence

**Workers Table** (created on plan selection):
```sql
INSERT INTO workers (
  phone, phone_number, platform, shift, 
  upi_id, pin_codes, language, plan,
  coverage_pct, is_active, last_seen_at
) VALUES (...)
```

**Policies Table** (week-based coverage):
```sql
INSERT INTO policies (
  worker_id, plan, status, week_start,
  coverage_pct, weekly_premium
) VALUES (...)
```

**Appeals Table** (NEW - for disputes):
```sql
INSERT INTO appeals (
  worker_id, phone, reason, status,
  created_at, expires_at
) VALUES (...)
```

## Error Handling

All handlers include try-except blocks:
- **Supabase errors** → "⚠️ Error during registration: {error}"
- **Invalid input** → Reprompt with error message + format example
- **Missing state** → Return HELP (default fallback)
- **Timeout** → Redis expires state after 1 hour
